# 合同管理模块重写设计文档

## 1. 背景与目标

当前 `safety-bms` 的合同模块围绕“草稿 -> 审核 -> 生效 -> 在线签订 -> 履行”的流程设计，已经包含审核流转、在线签字、最终 PDF 生成等能力。对于当前业务场景，这套流程偏重，存在以下问题：

- 业务人员高频操作链路过长，创建合同后还需要经过审核和在线签订两个并非刚需的中间环节。
- 前端页面同时承载列表、审核、签订、详情等多种职责，操作主线不够清晰。
- 文件语义混杂，系统草稿、标准草稿、签后文件等概念分散在多个字段中，不利于追溯。
- 当前流程与实际线下签约方式不完全一致，增加了培训和维护成本。

本次重写目标是将合同模块改造成“制单 + 线下签约 + 归档”的轻量化流程：

- 移除审核和在线签订能力。
- 保留完整合同业务信息录入，包括收款计划、服务周期、备注等字段。
- 使用统一模板生成可打印的合同草稿。
- 线下签署完成后，将扫描件或 PDF 作为附件上传并归档。
- 同时保留“系统生成草稿”和“已签附件”两类文件，方便追溯。
- 尽量复用现有合同主表及其与发票、收款、服务工单的关联，控制重写范围。

## 2. 范围

### 2.1 本次包含

- 重写前后端合同管理主流程。
- 收敛合同状态机为线下签约模式。
- 重构合同列表、编辑抽屉、详情抽屉的交互结构。
- 引入统一模板的合同草稿生成能力。
- 引入签后附件上传与归档能力。
- 调整依赖合同状态的下游业务规则。
- 处理历史合同数据和旧流程文件的兼容展示。

### 2.2 本次不包含

- 多模板合同体系。
- 审核流、审批意见、驳回原因等能力。
- 在线签字、签名图片、在线合成 PDF 等能力。
- 独立的合同 V2 表体系。

## 3. 目标用户与核心场景

目标用户为后台业务人员，核心任务是快速完成合同建档、打印、线下签约和后续归档。

关键场景：

1. 录入合同基础信息与商务信息，保存为草稿。
2. 生成统一模板的合同草稿，用于打印和线下盖章签字。
3. 在线下签好后，将扫描件或 PDF 上传到系统中，形成正式归档。
4. 基于合同实际履约进度管理“已签订 / 履行中 / 已完成 / 已终止”状态。
5. 在详情中同时查看系统草稿和已签附件，支持审计和追溯。

## 4. 业务流程设计

新流程收敛为四步：

1. 业务人员填写合同表单并保存，合同处于 `草稿`。
2. 在 `草稿` 状态下，用户可按需生成或重新生成系统合同草稿文件，用于打印。
3. 线下签订完成后，用户上传签后的扫描件或 PDF，上传成功后合同进入 `已签订`。
4. 按履约进度将合同推进至 `履行中`、`已完成`，必要时可进入 `已终止`。

### 4.1 关键原则

- 合同表单保存与草稿生成解耦。保存合同不会自动覆盖已生成草稿，草稿生成由用户主动触发。
- 系统草稿和签后附件必须作为两类文件分开存储、分开展示。
- 上传已签附件成功后，系统默认自动将合同置为 `已签订`，以匹配实际线下流程。
- 合同编号在创建时生成并固定，避免打印、签署、归档过程中的编号不一致。

## 5. 状态机设计

### 5.1 新状态集合

- `draft`：草稿
- `signed`：已签订
- `executing`：履行中
- `completed`：已完成
- `terminated`：已终止

### 5.2 允许流转

- `草稿 -> 已签订`
- `已签订 -> 履行中`
- `履行中 -> 已完成`
- `草稿 -> 已终止`
- `已签订 -> 已终止`
- `履行中 -> 已终止`

### 5.3 关键约束

- 没有至少一份 `signed` 类型附件时，不允许合同进入 `已签订`。
- `已完成` 前若仍存在未完成或未验收的服务工单，沿用当前校验逻辑进行拦截。
- 删除附件若会导致当前状态不合法，应阻止删除或要求先回退状态。
- `已完成` 和 `已终止` 状态下不允许再编辑核心业务字段，仅允许查看和必要的文件下载。

## 6. 前端交互设计

前端保持“合同模块”单一入口，但将职责重新划分为列表、编辑抽屉、详情抽屉三部分。

### 6.1 合同列表页

列表页作为主工作台，目标是让业务人员在一个页面中完成大部分高频操作。

建议保留的字段：

- 合同编号
- 合同名称
- 客户
- 合同金额
- 收款计划
- 服务周期
- 状态
- 最近更新时间

建议增强的筛选：

- 关键词（合同编号/名称）
- 状态
- 客户
- 签订日期范围
- 服务周期范围

按状态显示的行操作：

- `草稿`：编辑、生成草稿、上传已签附件、终止
- `已签订`：查看草稿、查看已签附件、开始履行、终止
- `履行中`：查看文件、标记完成、终止
- `已完成` / `已终止`：查看详情、下载文件

设计要点：

- 列表页负责推进流程，而不是承载复杂审批动作。
- 文件类操作优先就近完成，减少反复进入详情页的次数。

### 6.2 合同编辑抽屉

继续使用抽屉交互，保持与现有后台风格一致。表单按业务语义分组：

- 基本信息：合同编号、合同名称、客户
- 商务信息：合同金额、收款计划
- 履约信息：签订日期、服务开始日期、服务结束日期
- 补充信息：备注

交互原则：

- 保存后合同保持 `草稿`。
- 草稿生成作为独立按钮，不与保存合并，避免用户每次微调都覆盖文件。
- 创建和编辑尽量复用同一套表单组件，减少逻辑分叉。

### 6.3 合同详情抽屉

详情页改为“归档信息总览”，不再承载审核和在线签订流程。

分为三个区域：

1. 合同业务信息
2. 文件资料区
3. 过程记录区

文件资料区明确分成两组：

- `合同草稿`
  - 显示最新生成时间
  - 提供下载和打印入口
- `已签附件`
  - 显示上传时间、上传人、文件名
  - 提供下载或预览入口

过程记录区至少展示：

- 创建时间
- 最后修改时间
- 签后附件上传时间
- 状态变更记录

## 7. 数据模型设计

### 7.1 合同主表 `contracts`

继续复用 `contracts` 主表作为业务主体，保留以下核心字段：

- `contract_no`
- `title`
- `customer_id`
- `service_type`
- `total_amount`
- `payment_plan`
- `status`
- `sign_date`
- `start_date`
- `end_date`
- `remark`
- `created_by`

字段语义调整建议：

- `draft_doc_url`：保留为“当前最新系统草稿文件路径”。
- `signed_at`：保留，语义调整为“签后附件确认归档时间”。
- `template_id`：不再作为新主流程核心字段，可在过渡期保留以兼容旧数据和旧代码。
- `standard_doc_url`：与 `draft_doc_url` 存在概念重叠，重写后应统一为单一“系统草稿”语义，并逐步废弃重复字段。
- `final_pdf_url`：不再承载在线签署最终 PDF 语义，后续以附件表为准，不建议继续作为新流程主字段。

### 7.2 合同附件表 `contract_attachments`

新增合同附件表，用于统一管理合同相关文件。

建议字段：

- `id`
- `contract_id`
- `file_name`
- `file_url`
- `file_type`
- `uploaded_by`
- `uploaded_at`
- `remark`

`file_type` 取值：

- `draft`：系统生成草稿
- `signed`：线下签后附件
- `other`：保留扩展能力

设计原因：

- 支持一份合同保留多次草稿生成历史。
- 支持一份合同归档多份签后附件。
- 文件追溯逻辑从合同主表解耦，避免主表继续堆积文件语义。

### 7.3 历史记录

若当前已有 `ContractChange`，建议继续复用，用于记录：

- 状态变化
- 关键字段编辑摘要
- 文件上传动作摘要

这样无需额外新建审计表，也能满足详情中的过程记录展示。

## 8. 接口设计调整

接口整体从“审批签署系统”收敛为“制单归档系统”。

### 8.1 保留并调整的接口能力

- 创建合同
- 编辑合同
- 查询合同列表
- 查询合同详情
- 更新合同状态
- 生成合同草稿

### 8.2 新增或重构的接口能力

- 上传已签附件
- 查询合同附件列表
- 删除合同附件（需带状态合法性校验）

### 8.3 下线的接口语义

- 提交审核
- 审核通过/驳回
- 在线签订
- 上传在线签署结果
- 下载在线签署最终 PDF 的专属流程接口

### 8.4 关键业务规则

- 合同创建后默认 `draft`。
- 上传 `signed` 类型附件成功后，系统自动将合同流转为 `signed`。
- 手动变更为 `signed` 时必须校验已存在签后附件。
- 发票、收款等依赖合同状态的业务，应改为以 `signed` 及以上状态作为可用前提，不再依赖旧的 `active`。

## 9. 统一模板与文件策略

本次仅支持“一套模板覆盖大多数合同”。

策略如下：

- 使用统一 Word 模板生成合同草稿。
- 草稿文件始终视为系统生成产物，可多次重新生成。
- 签后扫描件或 PDF 视为正式归档件，不与草稿互相覆盖。
- 详情页默认展示最新草稿和最新签后文件，同时可查看历史附件。

文件存储策略：

- MinIO 继续作为底层对象存储。
- 草稿与签后附件在对象命名或目录上明确区分，避免混用。
- 下载地址仍通过预签名 URL 提供。

## 10. 错误处理设计

错误提示应面向业务人员，避免暴露内部实现细节。

### 10.1 草稿生成失败

- 不影响合同保存。
- 提示“合同草稿生成失败，请稍后重试”。
- 保留重新生成入口。

### 10.2 已签附件上传失败

- 合同保持原状态。
- 不写入半完成状态。
- 提示“已签合同上传失败，请重试”。

### 10.3 非法状态流转

- 无签后附件时进入 `已签订`：提示“请先上传已签合同附件”。
- 未完成工单时进入 `已完成`：沿用当前提示。
- 删除附件导致状态不合法：提示用户先调整状态或保留至少一份签后附件。

## 11. 迁移策略

### 11.1 分支策略

- 先将当前合同模块作为独立历史分支保留。
- 在 `main` 上进行合同模块重写。

### 11.2 数据迁移原则

旧状态到新状态的建议映射：

- `draft -> draft`
- `review -> draft`
- `active -> draft` 或 `signed`
- `signed -> signed`
- `executing -> executing`
- `completed -> completed`
- `terminated -> terminated`

其中 `active` 需要依据历史数据中是否已存在有效签后文件来决定：

- 若已有可视为正式版本的签后文件，则迁为 `signed`
- 否则迁为 `draft`

### 11.3 历史文件迁移

- 旧的系统草稿、标准草稿统一迁入 `draft` 类型附件。
- 旧的最终签署文件若存在，迁入 `signed` 类型附件。
- 合同详情页基于新附件模型展示历史文件，前端不再区分旧字段来源。

### 11.4 兼容期处理

- 旧字段在短期内可保留以降低迁移风险。
- 新前端页面不再直接依赖旧审核和在线签订字段。
- 迁移完成后，再评估清理废弃字段和接口。

## 12. 对下游模块的影响

### 12.1 发票与收款

- 允许引用合同的状态判断从旧 `active` 调整为新 `signed` 及以上。
- 金额校验逻辑不变，继续围绕合同总额、已开票金额、已收款金额执行。

### 12.2 服务工单

- 合同完成前仍需校验是否存在未完成或未验收工单。
- 若服务工单创建依赖合同状态，应同步适配新状态集。

### 12.3 导出与统计

- 合同导出、仪表盘、统计报表需要更新状态映射，移除旧状态文案。
- 若当前统计逻辑依赖 `active`，需改为依赖 `signed` 或 `executing`，按业务口径重新确认。

## 13. 验收标准

- 可以创建并保存包含收款计划、服务周期、备注在内的完整合同草稿。
- 草稿状态下可以成功生成、下载和打印统一模板的合同草稿。
- 上传签后附件后，合同自动进入 `已签订`。
- 合同可按 `已签订 -> 履行中 -> 已完成` 正常流转。
- 没有签后附件时，不允许进入 `已签订`。
- 合同详情页能同时展示系统草稿和已签附件。
- 历史合同在迁移后可继续查看原有草稿和正式文件。
- 发票、收款、服务工单等依赖合同状态的功能适配新状态后正常工作。
- 列表筛选、导出、详情查看均支持新状态体系。

## 14. 风险与控制

| 风险 | 说明 | 控制措施 |
| --- | --- | --- |
| 旧状态迁移不准确 | `active` 状态可能无法直接判断是否已完成线下签订 | 迁移脚本中加入文件存在性判断，并预留人工校验名单 |
| 旧字段与新附件模型并存导致逻辑混乱 | 过渡期前后端可能混用字段 | 新前端只消费新语义接口，后端在聚合层统一输出 |
| 下游模块仍依赖旧 `active` 状态 | 发票、收款、服务工单可能出现行为回归 | 在实现计划中列出所有状态依赖点并逐一替换验证 |
| 文件删除破坏状态一致性 | 删除最后一份签后附件后合同仍显示已签订 | 在删除接口中加入状态合法性校验 |

## 15. 实施建议

建议按以下顺序实施：

1. 先完成状态机、数据模型和接口语义收敛。
2. 再重写前端页面结构和交互。
3. 最后处理历史数据迁移、旧文件兼容和下游状态适配。

这样可以先稳定合同主线业务，再逐步清理兼容细节，降低一次性改动风险。
